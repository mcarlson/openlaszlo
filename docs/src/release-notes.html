Release Notes for OpenLaszlo @VERSIONID@

OpenLaszlo @VERSIONID@ is a bug fix release, containing more than 90 bug fixes in areas such as context menus, scrollbars, tab panes, mouse events, multiline text views, inputtext, DHTML/IE7 anomalies, iframes, and the history mechanism (which now works in SWF9 and SWF10). For details, see the JIRA report for Fixed Bugs in OpenLaszlo @VERSIONID@.

In addition, OpenLaszlo @VERSIONID@ contains new features: preliminary support for HTML 5 shadow across runtimes, compilation support for binary libraries across runtimes, support for the Google Chrome plug-in for DHTML/IE6, a simple mechanism for making <attribute>s CSS styleable, and Flash Player 10 support (SWF10). More details are provided below.

HTML 5 Shadow Across Runtimes

Preliminary support has been added for HTML 5 shadow across runtimes. See the OpenLaszlo <view> for details.

Binary Library Support

A binary library is a partially-compiled smaller library (an .lzo) that your application can link against. Because the binary library is smaller and partially compiled, linking and compilation are faster. To create a binary library, you compile the library as follows: 

lzc -c library.lzx

This creates a library.lzo file in the same directory. Any program that includes the library.lzx will prefer the .lzo if that is found. Your build process needs to ensure that the .lzo file is kept current.

Google Chrome Plug-in

Support for the Google Chrome plug-in has been been added to OpenLaszlo @VERSIONID@. Loading an OpenLaszlo DHTML application into IE6 will take you to a page where you can easily download and install the Google Chrome plug-in. Once the page refreshes, the application runs in Google Chrome.

<attribute> now CSS Styleable

In OpenLaszlo @VERSIONID@, there is a new style property of <attribute>, which is a string that can be used to associate a CSS property name with an attribute to make it easy to style your views with CSS. For example: 

  <view id="myView">
     <attribute name="bgcolor" ... style="background-color"/>
  </view>

This code associates the CSS background-color property with the view's bgcolor attribute. Thus the following CSS could be used to style the view's bgcolor: 

  <stylesheet>
    #myView { background-color: blue; }
  </stylesheet>

This mechanism is a more streamlined way of saying: 

  <attribute name="bgcolor" value="$style{'background-color}" />

But also allows you to define a fallback default for when there is no CSS style: 

  <attribute name="bgcolor" style="background-color" value="null" />

if there is no matching CSS specification, the bgcolor will take on the default value of null (meaning transparent).

SWF10 Support

The Dev Console now provides runtime target choices of SWF8, DHTML, and SWF10 (instead of SWF9). We strongly recommend using SWF10 because of the large number of improvements made to the AS3 libraries.

If you need to do so, you can still compile your application to SWF9 via the LPS server with explicit lzr=swf9 in the query string, or from the command line as lzc --runtime=swf9.

Operating System, Browser, Runtime Compatibility

This release has been fully tested on the following operating systems/browser/runtime platforms:

  • Windows XP, Internet Explorer 7: SWF8, SWF10, DHTML
  • Windows XP, Firefox 3.5: SWF8, SWF10, DHTML
  • Linux 2.6 kernel, Firefox 3.5: SWF8, SWF10, DHTML
  • Mac OS X, Firefox 3.5: SWF8, SWF10, DHTML
  • Mac OS X, Safari 4: SWF8, SWF10, DHTML

Please note that although we have also done some testing with Windows Vista, OpenLaszlo @VERSIONID@ has not been fully qualified against that operating system at this time. Please see Running OpenLaszlo Server on Windows Vista for more information.

Migrating your Applications to OpenLaszlo @VERSIONID@

For those of you who have already upgraded your applications to OpenLaszlo 4.2 or higher, no further work is needed. You should just start using OpenLaszlo @VERSIONID@.

To migrate your 4.0.X or 4.1.1 applications, we strongly suggest that you refer to this wiki page: Runtime_Differences. This page discusses the changes required by SWF9 and also provides a methodology for upgrading your application. It is very important that you run the automated conversion scripts in the recommended order, should you choose to take advantage of them. To review the changes in incremental releases, please refer to the OpenLaszlo Archive

Additional Notes

Some of the notable bug fixes in OpenLaszlo @VERSIONID@ are described below.

LPP-8372 Cursor attribute does not work

A pair of attributes has been added to the <resource> tag, offsetx and offsety. The purpose is to allow you to specify an offset of a bitmap resource used for a custom cursor, so that the mouse will align with the proper 'hotspot' in the cursor image.

Example: set a custom crosshair cursor from a bitmap resource, with an offset of (-8,-8) as follows: 

  <resource name="crosshair" src="cursor_crosshair.png" offsetx="-8" offsety="-8" />
  <view height="40" bgcolor="#cccccc" width="100" cursor="crosshair" />

The cursor offset feature currently only works in the SWF10 runtime, but will eventually work in DHTML and SWF8.

LPP-8479 Debug.evalCarefully not necessary after LPP-8222

When debugging is on, OpenLaszlo @VERSIONID@ catches, reports, and ignores errors that are not specifically declared with #pragma "throwsError=true". NOTE: If backtracing is off, the line numbers reported with errors are only accurate to the function that caused the error. For more accurate line numbers you must enable backtracing.

The debugger now indicates when its file/line information is only approximate. If the location information is derived from a view associated with the error, rather than @file... it will print %file... If the line number is only approximate (backtracing is not enabled), rather than #line it will print ~line.

Backtracing must be enabled in the developer console to get the most accurate file/line information in debugger messages, warnings, and errors.

LPP-8481 Debug.foo() should show the debugger window again once it's been closed

It is now possible to specify the following in your application to hide the debugger window until there is actually debugger output: 

  <debug visible="false" />

The debugger window will automatically make itself visible when there is any output to the debugger (including any runtime warnings or errors).

If you close the debugger window, it will re-open the next time there is output.

LPP-2207 Problems in Setup SOLO Application Deployment and Preview SOLO Application in Browser

"Setup SOLO Application Deployment" and "Preview SOLO Application in Browser" exhibit some minor cosmetic defects that we will not be fixing.

LPP-7070 Methods to determine if a view is in front of or behind another view

OpenLaszlo @VERSIONID@ adds these methods to check if one view is in front of or behind another view with the same parent: 

  LzView.isInFrontOf(sibling:View):boolean
  LzView.isBehind(sibling:View):boolean
LPP-8313 Support for downloadable fonts in DHTML (@font-face inclusion of OpenType or TTF fonts)

DHTML now supports font embedding for browsers that support it, including Firefox 3.5, Safari 3.1 and IE 7 and higher. Be sure to include an .otf version of the font alongside the .ttf for support in IE.

LPP-7176 SWF9: Make Debug.bugReport work in SWF9

Debug.bugReport now works correctly for AS3 platforms (SWF9 and SWF10).

LPP-8596 Add clipPath() support for drawviews in SWF9 and SWF10

drawview.clipPath() and clipButton() are now supported in SWF8 and SWF10.

LPP-8550 Standardized cross-runtime API to access in-detail browser information

lz.embed.browser is now available in the Flash runtimes.

Workaround for Input Text Unexpected Behavior with onfocus Handler

In some cases, the onfocus event fires too early, causing unexpected input text characters to be selected as it does in this example: 

  <canvas layout="axis: y">
    <inputtext text="abcd" width="100" bgcolor="0xcccccc"/>
    <inputtext text="abcd" width="100" bgcolor="0xffcccc" onfocus="this.setSelection(0, 1)"/>
   </canvas>

To work around this issue, you can create a cover view that handles initial clicks, as shown here for the above example: 

  <canvas layout="axis: y">
    <inputtext text="abcd" width="100" bgcolor="0xcccccc"/>
    <view bgcolor="0xffcccc"> <inputtext name="field" text="abcd" width="100">
      <handler name="onfocus"<
        parent.cover.setVisible(false);
        this.setSelection(0, this.getText().length);
      </handler>
      <handler name="onblur">
        parent.cover.setVisible(true);
      </handler>
      </inputtext>
      <view name="cover" width="100%" height="100%" onclick="LzFocus.setFocus(parent.field)" />
    </view>
  </canvas>

Deprecation Notice: Attribute Values on Data Element Will Be Coerced to String in a Future OpenLaszlo Release

Currently, if you set an attribute value for a data element, the value is stored as it is set. This will no longer be true in future OpenLaszlo releases, where the value will be coerced to a String value. If you need to set non-String values for data elements, you should use the setUserData method as shown in this example where node is an LzDataElement:
node1.setUserData('key1', val1); assertEquals(val1, node1.getUserData('key1'), 'getUserData key1')
